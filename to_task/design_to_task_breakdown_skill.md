
# Design → Task Breakdown Skill（AI-Optimized）

## 概要
設計から、実装・テスト・リリース可能なタスクへ分解するための skill。  
AI活用を前提とし、再現性のあるタスク分解を目的とする。

---

## 目的
- 設計を実装可能な単位に分解する
- タスク粒度を統一する
- 依存関係を明確にする
- 抜け漏れを防ぐ
- テスト・リリースまで含める

---

## 入力
- 設計ドキュメント（Requirement → Designの出力）

---

## 出力（必須）
- タスク一覧
  - backend
  - frontend
  - data / migration
  - infra
  - test
  - release
- 各タスク詳細
  - 目的
  - 内容
  - 入出力
  - 依存関係
- リスク

---

## 基本原則
- 1タスク = 1責務
- 実装可能な粒度まで分解する
- 抽象タスクは禁止
- テスト・リリースを含める
- UI / API / データの整合を保つ
- 正常系・異常系を分ける

---

# Step1 機能分解

## 出力
- 機能単位のタスクブロック

## 観点
- 独立しているか
- 依存関係はあるか

---

# Step2 層分解

## 出力
- backend / frontend / data / infra

## 観点
- 各層に分解されているか
- APIとUIが対応しているか

---

# Step3 backend

## 出力
- API実装
- ロジック
- バリデーション
- エラー処理

## 観点
- 入出力
- 状態変更
- 権限
- 異常系

---

# Step4 frontend

## 出力
- UI
- コンポーネント
- 状態管理
- API連携

## 観点
- 入力
- エラー表示
- 状態管理

---

# Step5 データ

## 出力
- テーブル / カラム
- インデックス
- migration

## 観点
- 既存影響
- rollback可能か

---

# Step6 infra

## 出力
- 環境設定
- 外部連携
- ログ

## 観点
- セキュリティ
- 障害対応

---

# Step7 テスト

## 出力
- unit / integration / e2e

## 観点
- 正常系
- 異常系
- 境界値

---

# Step8 リリース

## 出力
- デプロイ
- migration
- rollback

## 観点
- 無停止可能か

---

# Step9 依存関係

## 出力
- タスク依存

## 観点
- 順序
- 並列可否

---

# Step10 リスク

## 出力
- リスク一覧

## 観点
- 技術
- データ
- 外部依存

---

## チェックリスト
- 全機能がタスク化されているか
- 各層が揃っているか
- テストがあるか
- リリースがあるか
- 抽象タスクがないか

---

## NGパターン
- 曖昧タスク（例：API作る）
- テストなし
- 依存関係なし

---

## 本質
設計を「作業」に変換すること
