
# Code → Performance Risk Check Skill（AI-Optimized）

## 概要
実装コードまたは変更差分を入力として、性能上のリスクや将来的なボトルネック候補を検出するための skill。
AI活用を前提とし、「今は動くが、負荷時に危険な実装」を早期に見つけることを目的とする。

---

## 目的
- 性能事故を未然に防ぐ
- ボトルネック候補を早期に発見する
- 負荷増加時に危険な実装を洗い出す
- パフォーマンス観点のレビューを標準化する
- 改善優先度を明確にする

---

## 入力
- 実装コード
- 変更差分（diff）
- Implementation Plan（任意）
- タスク内容（任意）
- クエリ / API仕様 / データ量前提（任意）
- 既存性能要件（任意）

---

## 出力（必須）
- 性能リスク一覧
- 発生条件
- 影響範囲
- リスクレベル（高 / 中 / 低）
- 改善提案
- 追加で必要な計測 / 検証項目

---

## 基本原則
- 現時点で問題化していなくても、負荷時のリスクを見る
- 局所最適ではなく全体の処理フローで考える
- 推測で安全判定しない
- データ量・同時実行・外部依存を前提に考える
- 「遅いかもしれない」ではなく、発生条件まで具体化する
- 計測が必要なものは計測前提で扱う

---

# Step1 実行経路の把握

## 出力
- 主な処理フロー
- 重そうな経路候補

## 観点
- ユーザー操作起点か
- バッチ / 非同期処理か
- 同期で待たせる処理か
- 高頻度で呼ばれる処理か

---

# Step2 計算量リスクの検出

## 出力
- 計算量リスク一覧

## 観点
- ネストしたループがあるか
- データ件数に比例して重くなるか
- 毎回全件走査していないか
- 不要なソート / フィルタ / 集計がないか

---

# Step3 DB / クエリリスクの検出

## 出力
- DBアクセスリスク一覧

## 観点
- N+1 の可能性
- ループ内クエリ
- フィルタ条件不足
- インデックス前提か
- 件数増加時に遅くなるクエリか
- 不要なSELECT *がないか
- OFFSET依存が大きすぎないか

---

# Step4 API / 外部依存リスクの検出

## 出力
- 外部依存性能リスク一覧

## 観点
- 同期で外部APIを待っていないか
- 連続リクエストになっていないか
- リトライが過剰でないか
- タイムアウト設計があるか
- キャッシュ余地があるか

---

# Step5 メモリ使用リスクの検出

## 出力
- メモリリスク一覧

## 観点
- 大量データを一括読み込みしていないか
- 不要なコピーが多くないか
- 巨大オブジェクトを保持し続けないか
- ストリーム処理 / 分割処理が必要ではないか

---

# Step6 同時実行リスクの検出

## 出力
- 並行処理リスク一覧

## 観点
- ロック競合が起きないか
- 同時リクエストでボトルネック化しないか
- 同一リソース争奪が起きないか
- キュー詰まりが起きないか
- goroutine / thread を増やしすぎていないか

---

# Step7 UI / フロントエンド性能リスクの検出

## 出力
- UI性能リスク一覧

## 観点
- 不要な再レンダリングがないか
- 大量DOM / 大量描画をしていないか
- 逐次API呼び出しになっていないか
- 初回表示が重くならないか
- キャッシュ / メモ化余地があるか

---

# Step8 監視・計測不足の検出

## 出力
- 計測不足項目
- 追加すべき観測ポイント

## 観点
- 処理時間を測れているか
- 失敗率を追えるか
- ボトルネック調査に必要なログ / メトリクスがあるか
- p95 / p99 で見るべき処理か

---

# Step9 改善優先度の整理

## 出力
- 高 / 中 / 低リスク分類
- 優先順位付き改善提案

## 観点
- 発生頻度
- ユーザー影響
- データ量増加耐性
- 修正コスト
- 緊急性

---

## チェックリスト
- 高頻度処理の重さを見ているか
- DBアクセスの問題を見ているか
- 外部依存の待ち時間を見ているか
- メモリ使用量の問題を見ているか
- 同時実行時の問題を見ているか
- 監視 / 計測不足を見ているか
- 発生条件まで具体化されているか

---

## NGパターン
- 「今は遅くない」で終わる
- 計算量や件数前提を見ない
- DB / 外部依存を無視する
- 抽象的に「重そう」と言うだけ
- 改善提案や計測提案を出さない

## 本質
性能問題は「今遅いかどうか」ではなく、
**データ量・同時実行・外部依存が増えたときに破綻する構造を先に見つけること** が本質である。
