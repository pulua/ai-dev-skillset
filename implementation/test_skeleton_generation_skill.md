
# Code → Test Skeleton Generation Skill（AI-Optimized）

## 概要
実装コードまたは変更差分を入力として、テストコードの雛形を生成するための skill。
AI活用を前提とし、「テストケースを実際に書き始められる形に変換すること」を目的とする。

---

## 目的
- テスト実装の初速を上げる
- テストコードの構造を統一する
- テストケースをコードに落とし込む
- モック対象・前提データ・アサーション観点を明確にする
- テスト漏れを減らす

---

## 入力
- 実装コード
- 変更差分（diff）
- テストケース一覧（任意）
- タスク内容（任意）
- 使用テストフレームワーク / ライブラリ
- 既存テストコード（任意）

---

## 出力（必須）
- テストコードの雛形
- テスト関数名
- テストケース一覧
- 前提データ
- モック対象
- アサーション候補
- 未確定事項

---

## 基本原則
- 実装言語・既存テストスタイルに合わせる
- いきなり完成版テストを目指さず、書き始められる雛形を出す
- 正常系だけで終わらない
- モックが必要な依存を明示する
- テストデータの前提を明示する
- 1テスト = 1意図を意識する
- テーブル駆動テストが適する場合は優先する
- 実行しやすい単位で出力する

---

# Step1 テスト対象の特定

## 出力
- テスト対象関数 / メソッド / コンポーネント
- テスト単位

## 観点
- 何をテストするのか
- 単体テストか結合テストか
- 関数単位かAPI単位かUI単位か
- 既存テストを拡張するか新規作成するか

---

# Step2 テスト種別の整理

## 出力
- unit test
- integration test
- e2e test
- component test

## 観点
- 最小コストで担保できるテスト種別は何か
- 外部依存をモックすべきか実呼び出しすべきか
- ロジック中心かUI中心か
- どこまでを責務として検証するか

---

# Step3 テストケースのコード化

## 出力
- テストケース一覧
- テスト関数の雛形

## 観点
- 正常系
- 異常系
- 境界値
- 状態遷移
- 権限
- 外部依存
- 競合 / 重複

---

# Step4 前提データの整理

## 出力
- テストデータ
- 初期状態
- DB fixture / stubデータ

## 観点
- テストに必要な最小データは何か
- 既存データ前提があるか
- 状態の初期化が必要か
- テストごとに独立性を保てるか

---

# Step5 モック対象の整理

## 出力
- モック対象一覧
- モックの返り値パターン
- スタブの前提

## 観点
- 外部API
- DB repository
- queue / cache
- 時刻依存
- UUID生成
- 認証 / 認可依存

---

# Step6 アサーション観点の整理

## 出力
- アサーション候補一覧

## 観点
- 戻り値
- エラー
- DB更新結果
- 呼び出し回数
- 状態変更
- UI表示
- 副作用（通知 / イベント / ログ）

---

# Step7 テスト構造の決定

## 出力
- テストファイル構成
- describe / context / table test 構造
- helper関数候補

## 観点
- 既存テストスタイルに沿っているか
- 重複を減らせるか
- ケース追加しやすいか
- 可読性があるか

---

# Step8 雛形コード生成

## 出力
- 実際のテストコード雛形

## 観点
- すぐ追記できる形か
- TODOの粒度が適切か
- コンパイル可能に近いか
- 不足箇所が明示されているか

---

# Step9 未確定事項の明示

## 出力
- 未確定事項一覧

## 観点
- 正しい期待結果が仕様から読み取れない箇所はあるか
- 既存のテスト方針が不明な箇所はあるか
- モック方法が確定できない箇所はあるか

---

## チェックリスト
- テスト対象が明確か
- テスト種別が適切か
- 正常系 / 異常系 / 境界値が含まれているか
- モック対象が整理されているか
- 前提データが明確か
- アサーション観点があるか
- 既存テストスタイルに沿っているか
- すぐ実装を始められる雛形になっているか

---

## NGパターン
- 正常系の雛形しか出さない
- モック対象を無視する
- 前提データが不明
- アサーションが曖昧
- 実行不能な擬似コードだけ出す
- 既存テストスタイルを無視する

## 本質
テストケースを「読むもの」から「すぐ書けるもの」に変換すること。
重要なのは、網羅性だけでなく、実際に実装へ着手しやすいテスト構造を作ることである。
